Intuition First

Before writing a single line of SQL — understand the why.

🎓

The Real-World Problem

"I want to find all students who scored above the class average. But I don't know the average yet — I need to calculate it first, then use it."

Why not just write two queries?

You could run SELECT AVG(score) first, note the number, then write a second query. But that's:

  • Error-prone (manual copy)
  • Not reusable or dynamic
  • Breaks on changing data
✅ Subquery solves this elegantly in ONE query
💡 Human Thinking → SQL Translation
1
Think: "What's the class average?" → SQL: SELECT AVG(score) FROM students
2
Think: "Who scored above that?" → SQL: WHERE score > [that average]
3
Combine: Put the first query inside the second — that's your subquery!
4
Result: One elegant query that is dynamic, reusable, and always up-to-date.
📌 Section 02

What Is a Subquery?

A query nested inside another query. The inner query runs first, its result is used by the outer query.

Inner Query

Runs first
Calculates something

→

Output / Result

A value, a list,
or a table

→

Outer Query

Uses the output
to filter / compute

📄 Basic Structure
-- Outer query uses the result of inner query SELECT name, score FROM students WHERE score > ( -- Inner query: calculates the class average SELECT AVG(score) FROM students );
🔢 Section 03

Types of Subqueries

Four key types — each with a different shape of output and purpose.

1️⃣

Scalar Subquery

Returns a single value (one row, one column). Used in WHERE, SELECT, HAVING.

Example
SELECT name, salary FROM employees WHERE salary > ( SELECT AVG(salary) FROM employees );
📋

Column Subquery

Returns a single column with multiple rows. Used with IN, ANY, ALL.

Example
SELECT name FROM employees WHERE dept_id IN ( SELECT id FROM departments WHERE location = 'NY' );
📊

Multiple Row Subquery

Returns multiple rows and columns — used as a derived table in FROM.

Example
SELECT dept, AVG(salary) FROM ( SELECT dept, salary FROM employees WHERE active = 1 ) AS active_emp GROUP BY dept;
🔍

EXISTS Subquery

Checks if subquery returns any rows. Returns TRUE/FALSE. Very efficient.

Example
SELECT name FROM customers c WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.cust_id = c.id );
🔥 Section 04

The Interviewer's Thinking Framework

This 4-step mental model is how top candidates crack subquery problems in real interviews.

1

What is the final result?

Define the output clearly. "I want employees + their salaries." This becomes your outer SELECT.

2

What intermediate data do I need?

Identify what you need to filter/compare against. "I need the average salary per department."

3

Can a subquery generate it?

Write the inner query in isolation. Test it mentally — does it return what you expect?

4

How will the outer query use it?

Connect the two: WHERE salary > (inner), IN (inner), EXISTS (inner), or FROM (inner) AS alias.

🧠 Apply the Framework: "Employees earning more than dept average"
1

2
Intermediate: Average salary for each employee's department → AVG(salary) WHERE dept_id = [same dept]
3
Subquery: Yes — correlated subquery where inner query references outer row's dept_id
4
Connect: WHERE e.salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id)
🔄 Section 05

Correlated Subqueries

The most powerful — and most misunderstood — type of subquery.

Normal Subquery

Runs ONCE. Returns result. Outer uses it.
-- Inner runs once SELECT name FROM students WHERE score > ( SELECT AVG(score) FROM students );
✅ Independent of outer query rows

Correlated Subquery

Runs FOR EACH ROW. References outer row's data.
-- Inner runs for EACH employee SELECT name FROM employees e WHERE salary > ( SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id -- ↑ references outer row! );
⚡ Depends on each outer row — runs N times

🔄 How Correlated Subquery Executes

👤
Row 1: Alice (Dept A) → Run inner: AVG(salary) WHERE dept_id = A
→
Result: 55,000
Alice 72k > 55k ✅
👤
Row 2: Bob (Dept B) → Run inner: AVG(salary) WHERE dept_id = B
→
Result: 80,000
Bob 60k < 80k ❌
👤
Row 3: Carol (Dept A) → Run inner: AVG(salary) WHERE dept_id = A
→
Result: 55,000
Carol 90k > 55k ✅
⚖️ Section 06

Subquery vs JOIN

Know when to use each — this is a classic interview question.

Dimension 🔹 Subquery 🟣 JOIN
Readability Often clearer for nested logic Clearer for combining tables
Use when... You need an intermediate result to compare against You need columns from both tables in output
Performance Can be slower (correlated = N runs) Usually faster with proper indexes
EXISTS check Prefer Subquery Use LEFT JOIN + IS NULL
Aggregation Prefer Subquery Needs GROUP BY, more complex
Columns needed Only from one table Prefer JOIN

🎤 Section 09

Interview Questions

Conceptual questions interviewers love. Click to reveal answers.

Q1. What is the difference between EXISTS and IN? When would you prefer one over the other?
+
IN evaluates the outer query column against a list of values returned by the subquery. It materializes the full result set first. EXISTS checks for the existence of any row matching the condition — it short-circuits as soon as one match is found.

Prefer EXISTS when: The subquery might return many rows, when you only care about existence (not values), and when the subquery columns might contain NULLs (IN with NULLs can produce unexpected results).

Prefer IN when: Working with a small, fixed list of values or a small subquery result set.
Q2. What is a correlated subquery and how does it differ from a non-correlated one?
+
A non-correlated subquery is independent — it executes once, returns a result, and the outer query uses that result. It can be extracted and run on its own.

A correlated subquery references a column from the outer query (using an alias). It re-executes for every row of the outer query, making it context-dependent and typically slower. Use correlated subqueries when the inner query result depends on each outer row (e.g., per-department averages).
Q3. Why can NOT IN produce unexpected results when the subquery contains NULLs?
+
In SQL, any comparison with NULL using = or != returns UNKNOWN (not TRUE or FALSE). When NOT IN is used and the subquery returns even one NULL value, the entire NOT IN expression evaluates to UNKNOWN for every row — meaning no rows are returned.

Fix: Use WHERE col_id NOT IN (SELECT col_id FROM table WHERE col_id IS NOT NULL), or switch to NOT EXISTS which handles NULLs correctly.
Q4. What are the performance implications of using a correlated subquery vs a JOIN?
+
A correlated subquery runs once per row in the outer query — O(N) inner executions. For a table with 1 million rows, that's 1 million inner query executions. A JOIN with proper indexing typically runs in O(N log N) or better.

However, modern query optimizers (MySQL 8+, PostgreSQL) can rewrite correlated subqueries as joins automatically. Always check EXPLAIN/EXPLAIN ANALYZE to verify. For large datasets, prefer CTEs or JOINs for better optimizer hints.
Q5. Can a subquery appear in the SELECT clause? What does it return?
+
Yes — a scalar subquery in the SELECT clause returns a single value (one row, one column) for each row of the outer query. It effectively computes a derived column.

Example: SELECT name, (SELECT COUNT(*) FROM orders WHERE cust_id = c.id) AS order_count FROM customers c

This is technically a correlated subquery in the SELECT clause. If it returns more than one row, SQL throws an error. Use aggregation (COUNT, MAX, etc.) to ensure scalar output.
⚠️ Section 10

Common Mistakes

Learn from these traps before your next interview.

⛔ Using IN with Nullable Columns

If the subquery result contains NULLs, NOT IN returns zero rows — silently and unexpectedly.

-- DANGEROUS: NULLs break NOT IN WHERE id NOT IN (SELECT mgr_id FROM emp) -- SAFE: filter NULLs first WHERE id NOT IN (SELECT mgr_id FROM emp WHERE mgr_id IS NOT NULL)

⛔ Scalar Subquery Returning Multiple Rows

If your WHERE clause expects one value but the subquery returns multiple rows, SQL throws a runtime error.

-- ERROR: returns multiple rows WHERE salary = (SELECT salary FROM emp WHERE dept='IT') -- FIX: use IN or add aggregation WHERE salary IN (SELECT salary FROM emp WHERE dept='IT')

⛔ Forgetting Table Alias in Correlated Subquery

Without an alias, the inner query can reference the wrong table — same table name confuses the engine.

-- WRONG: no alias, ambiguous reference FROM employees WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = dept_id) -- RIGHT: alias 'e' makes reference clear FROM employees e WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id)

⛔ Using Subquery When JOIN Is Better

If you need columns from both tables in SELECT, a subquery in WHERE won't work — you need a JOIN.

-- WRONG: can't SELECT dept name from subquery SELECT name, dept_name -- dept_name not in scope! FROM employees WHERE dept_id IN (SELECT id FROM departments) -- RIGHT: use JOIN to access dept_name SELECT e.name, d.dept_name FROM employees e JOIN departments d ON e.dept_id=d.id
