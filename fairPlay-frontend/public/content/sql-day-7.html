The Problem With Nested Queries

You've been asked to fix a bug. You open the query and see this…

❌ The Nested Nightmare
SQL — HARD TO READ
SELECT e.name, e.salary
FROM employees e
WHERE e.dept_id IN (
  SELECT dept_id
  FROM (
    SELECT dept_id,
           AVG(salary) avg_sal
    FROM employees
    GROUP BY dept_id
    HAVING AVG(salary) > 60000
  ) dept_avgs
) AND e.salary > (
  SELECT AVG(salary)
  FROM employees
  WHERE dept_id = e.dept_id
);
✅ CTE Version — Reads Like Prose
SQL — CLEAN CTE
WITH dept_avgs AS (
  SELECT dept_id,
         AVG(salary) avg_sal
  FROM employees
  GROUP BY dept_id
),
high_dept AS (
  SELECT dept_id
  FROM dept_avgs
  WHERE avg_sal > 60000
)
SELECT e.name, e.salary
FROM employees e
JOIN high_dept h
  ON e.dept_id = h.dept_id
JOIN dept_avgs d
  ON e.dept_id = d.dept_id
WHERE e.salary > d.avg_sal;

Core Concept

What Exactly is a CTE?

A Common Table Expression is a temporary, named result set defined using the WITH keyword. It exists only for the duration of one query execution.

Anatomy of a CTE
-- ① Define the CTE using WITH
WITH cte_name AS (
    SELECT column1, column2
    FROM some_table
    WHERE condition
)
-- ② Reference the CTE just like a table
SELECT *
FROM cte_name
WHERE column1 > 100;
1
Engine reads the WITH block first
SQL executes the CTE definition and holds the result as a named temporary view — scoped to this query only. Nothing written to disk.
2
Main query references CTE by name
You use cte_name exactly like a table in FROM, JOIN, or nested WHERE clauses.
3
CTE is discarded after query completes
Unlike temp tables, CTEs don't persist. They are scoped to a single query — clean, disposable, zero cleanup needed.
💡

Key Insight: CTEs are not stored in the database. They are named subqueries that SQL evaluates inline. Think of them as aliases for complex logic — not physical storage.

Chaining

Multiple CTEs — Pipeline Pattern

Define multiple CTEs separated by commas. Each can reference the ones before it — creating a data pipeline.

Multi-CTE Chain
WITH
-- Step 1: Aggregate base data
dept_stats AS (
    SELECT dept_id,
           AVG(salary) AS avg_sal,
           COUNT(*) AS headcount
    FROM employees
    GROUP BY dept_id
),
-- Step 2: Filter using Step 1
top_depts AS (
    SELECT dept_id
    FROM dept_stats       -- ← references CTE 1
    WHERE avg_sal > 70000
      AND headcount > 5
)
-- Final query uses both CTEs
SELECT e.name, e.salary, d.avg_sal
FROM employees e
JOIN top_depts t ON e.dept_id = t.dept_id
JOIN dept_stats d ON e.dept_id = d.dept_id;

Advanced

🌳 Recursive CTE — Intuition Only

A recursive CTE calls itself — perfect for hierarchical data like org charts, folder trees, or category paths. It has two parts: an anchor (starting point) and a recursive member (the repeating step).

Org Chart — Recursive
WITH RECURSIVE org_tree AS (
    -- Anchor: start from CEO (no manager)
    SELECT id, name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: find each person's reports
    SELECT e.id, e.name, e.manager_id,
           ot.level + 1
    FROM employees e
    JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT name, level FROM org_tree
ORDER BY level;
CEO (level 0) → VP (level 1) → Manager (level 2) → Engineer (level 3) → …
🔥 Interview Scenarios

6 Real Interview Problems

Step-by-step thinking for each problem — exactly how a senior engineer would approach it.

01
Employees Earning Above Department Average
CTE replaces a correlated subquery — classic pattern
🧠 Step-by-Step Thinking
Compute average salary per department — store in CTE
Join CTE back to employees table
Filter where individual salary exceeds department average
SQL
WITH dept_avg AS (
    SELECT dept_id,
           AVG(salary) AS avg_salary
    FROM employees
    GROUP BY dept_id
)
SELECT e.name, e.salary, d.avg_salary
FROM employees e
JOIN dept_avg d ON e.dept_id = d.dept_id
WHERE e.salary > d.avg_salary
ORDER BY e.dept_id, e.salary DESC;
💡

By separating aggregation into a CTE, we avoid a correlated subquery running for every row. Cleaner and often faster.

02
Top 3 Salaries Per Department
Window functions + CTE — a must-know combo
🧠 Step-by-Step Thinking
Use DENSE_RANK() to rank employees within each department by salary (descending)
Wrap the ranking inside a CTE — you can't filter on window alias directly
Outer query filters WHERE rnk ≤ 3
SQL
WITH ranked AS (
    SELECT
        name, dept_id, salary,
        DENSE_RANK() OVER (
            PARTITION BY dept_id
            ORDER BY salary DESC
        ) AS rnk
    FROM employees
)
SELECT name, dept_id, salary, rnk
FROM ranked
WHERE rnk <= 3
ORDER BY dept_id, rnk;
💡

You cannot filter on window function aliases in WHERE. The CTE wrapper is the solution — it makes the alias available for filtering.

03
Find Duplicate Records Using CTE
ROW_NUMBER + CTE for clean deduplication
🧠 Step-by-Step Thinking
Assign row numbers within groups of identical records (same email + name)
First occurrence gets row_num = 1, duplicates get 2, 3…
Filter WHERE row_num > 1 to view or delete duplicates
SQL
WITH dup_check AS (
    SELECT *,
        ROW_NUMBER() OVER (
            PARTITION BY email, name
            ORDER BY id
        ) AS row_num
    FROM customers
)
-- View duplicates
SELECT * FROM dup_check
WHERE row_num > 1;

-- Delete duplicates (keep first)
DELETE FROM customers
WHERE id IN (
    SELECT id FROM dup_check
    WHERE row_num > 1
);
04
Running Total Using CTE
Cumulative sum — essential for finance & sales reports
🧠 Step-by-Step Thinking
CTE 1: Aggregate total sales per day
CTE 2: Apply SUM() window function with ROWS UNBOUNDED PRECEDING
Final SELECT reads both — clean and debuggable
SQL
WITH
daily_sales AS (
    SELECT sale_date,
           SUM(amount) AS daily_total
    FROM orders
    GROUP BY sale_date
),
running_totals AS (
    SELECT
        sale_date, daily_total,
        SUM(daily_total) OVER (
            ORDER BY sale_date
            ROWS BETWEEN UNBOUNDED PRECEDING
            AND CURRENT ROW
        ) AS running_total
    FROM daily_sales
)
SELECT * FROM running_totals
ORDER BY sale_date;
05
Second Highest Salary Using CTE
Classic interview trap — solved cleanly
🧠 Step-by-Step Thinking
Use DENSE_RANK() — not ROW_NUMBER() — to handle salary ties correctly
Wrap ranking in CTE so we can filter on the alias
SELECT WHERE rnk = 2 from CTE
SQL
WITH salary_ranked AS (
    SELECT
        name, salary,
        DENSE_RANK() OVER (
            ORDER BY salary DESC
        ) AS rnk
    FROM employees
)
SELECT name, salary
FROM salary_ranked
WHERE rnk = 2;
💡

Use DENSE_RANK not ROW_NUMBER. If two people tie for first, ROW_NUMBER gives rank 1 and 2 — skipping the real 2nd. DENSE_RANK gives both rank 1 and starts from 2 correctly.

06
Convert Nested Query to Readable CTE Steps
Refactoring is a skill — practice it
🧠 Step-by-Step Thinking
Identify each nested subquery and give it a meaningful name
Extract them into CTEs ordered by dependency
Final SELECT becomes one clean, readable sentence
❌ Before (3 nesting levels)
SELECT name
FROM employees
WHERE dept_id IN (
  SELECT dept_id
  FROM departments
  WHERE location = 'NYC'
) AND salary > (
  SELECT AVG(salary)
  FROM employees
  WHERE dept_id IN (
    SELECT dept_id
    FROM departments
    WHERE location = 'NYC'
  )
);
✅ After — Named Steps
WITH
nyc_depts AS (
  SELECT dept_id
  FROM departments
  WHERE location = 'NYC'
),
nyc_avg AS (
  SELECT AVG(salary) avg_sal
  FROM employees
  WHERE dept_id IN
    (SELECT dept_id FROM nyc_depts)
)
SELECT name
FROM employees
WHERE dept_id IN
  (SELECT dept_id FROM nyc_depts)
AND salary >
  (SELECT avg_sal FROM nyc_avg);

Interview Prep

5 Conceptual Interview Questions

Asked in data engineering, analytics engineering, and backend interviews. Know these cold.

Q1 — What is the difference between a CTE and a Subquery?
A CTE is defined before the main query using WITH and can be referenced multiple times by name. A subquery is inline and must be repeated if reused. CTEs dramatically improve readability for complex logic. Performance is generally equivalent — the query optimizer may treat them identically — but CTEs win on maintainability every time.
Q2 — What is the difference between a CTE and a Temporary Table?
A CTE is scoped to a single query — it evaporates after the query runs with zero cleanup. A temp table is persisted to disk (temporarily) for the session duration, supports indexes, and can be queried multiple times. Use temp tables for large intermediate datasets or multi-query workflows where a CTE would recompute expensively.
Q3 — Can a CTE reference itself? How does it work?
Yes — that's a Recursive CTE using WITH RECURSIVE. It has an anchor member (base case) joined by UNION ALL to a recursive member that references the CTE itself. Used for hierarchical data: org trees, folder structures, graph traversal. Always define a termination condition to prevent infinite loops.
Q4 — When would you NOT use a CTE?
Avoid CTEs when: (1) the intermediate result is very large and used multiple times — a temp table with an index will be significantly faster. (2) You need the result across multiple queries in the same session. (3) You're in a system where CTE materialization behavior is uncertain. (4) The logic is trivially simple — adding a CTE wrapper is unnecessary ceremony.
Q5 — Are CTEs materialized or re-evaluated each time they're referenced?
This is engine-dependent. PostgreSQL (before v12) materializes CTEs — computes once, stores result. v12+ may inline them. MySQL and SQL Server typically inline them. This means a CTE referenced twice might be computed twice. When in doubt, check the execution plan with EXPLAIN ANALYZE — or use a temp table if you need guaranteed single-computation.
Watch Out

⚠️ 5 Common Mistakes

These trip up even experienced developers.

❌
Thinking CTEs store data permanently. CTEs are temporary — they exist only during query execution. Nothing is written to disk. Use temp tables if you need data across multiple queries.
❌
Overusing CTEs for tiny logic. If a subquery is two lines, wrapping it in a CTE is overkill. Use CTEs when they genuinely improve readability or enable reuse.
❌
Wrong ordering when chaining. A CTE can only reference CTEs defined before it. Defining cte2 before cte1 when cte2 depends on cte1 throws an error.
❌
Using ROW_NUMBER instead of DENSE_RANK for rankings. ROW_NUMBER assigns unique ranks even for ties. DENSE_RANK handles ties correctly — rank 1, 1, 2, 3 instead of 1, 2, 3, 4.
❌
Recursive CTE without termination condition. Without a proper stop condition (depth limit or NULL check), a recursive CTE can loop infinitely and crash your session.

The Decision Rule
-- When to use what?

IF logic is complex OR multi-step OR reused in same query:
    → Use CTE

IF condition is simple AND used once:
    → Use Subquery

IF data is large AND needed across multiple queries:
    → Use Temp Table

IF structure is hierarchical (tree, graph, org chart):
    → Use Recursive CTE
