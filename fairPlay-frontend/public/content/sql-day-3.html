Core Concept

GROUP BY

Splits rows into groups based on a column's value, then applies an aggregate function to each group independently.

SELECT city, COUNT(*) AS student_count
FROM Students
GROUP BY city;

📌 Multiple Column Grouping
-- Group by city AND department together
SELECT city, dept, COUNT(*) AS count, AVG(marks) AS avg_marks
FROM Students
GROUP BY city, dept;

-- Result: each unique (city, dept) pair becomes one row
⚠️ Rule: Every column in SELECT that is not inside an aggregate function must appear in GROUP BY.
Filter After Grouping

HAVING Clause

WHERE filters raw rows before grouping. HAVING filters the grouped results after aggregation — it can use aggregate functions directly.

SELECT city, COUNT(*) AS student_count
FROM Students
GROUP BY city
HAVING COUNT(*) > 2;

-- Only show cities that have MORE THAN 2 students

WHERE vs HAVING

The most common interview question on this topic. Understand the fundamental difference.

WHERE
FiltersIndividual rows
Works onRaw data (before grouping)
Aggregates❌ Cannot use
OrderExecutes 2nd (after FROM)
ExampleWHERE salary > 50000

FiltersGrouped results
Works onAggregated data (after GROUP BY)
Aggregates✅ Can use
OrderExecutes 4th (after GROUP BY)
ExampleHAVING AVG(salary) > 60000
💡 Example Showing Both Together
-- Find departments where average salary > 60,000
-- but only consider employees with experience > 2 years

SELECT dept, AVG(salary) AS avg_sal
FROM Employees
WHERE experience > 2 -- filters rows FIRST
GROUP BY dept
HAVING AVG(salary) > 60000 -- filters groups AFTER
ORDER BY avg_sal DESC;

Why does execution order matter?
This is why you can't use aliases defined in SELECT inside a WHERE clause — WHERE runs before SELECT! It's also why aggregate functions in WHERE cause errors — aggregation happens at GROUP BY step, not WHERE step. And it's why HAVING can filter on aggregates but WHERE cannot.

Interview Ready

Top Interview Questions

These come up in almost every SQL interview. Know them cold.

Q1What is the difference between WHERE and HAVING in SQL?
WHERE filters individual rows before grouping and cannot use aggregate functions like COUNT, SUM, AVG. HAVING filters after GROUP BY and is specifically designed to work with aggregate functions. If you try to use WHERE COUNT(*) > 2, you'll get an error — that condition belongs in HAVING.
Q2Can we use WHERE with aggregate functions? Why or why not?
No. Aggregate functions are computed after rows are grouped (at the GROUP BY stage), but WHERE executes before grouping. Since the aggregate value doesn't exist yet when WHERE is evaluated, using it causes an error. The correct approach is to use HAVING, which runs after GROUP BY. Exception: you can use a subquery — WHERE salary > (SELECT AVG(salary) FROM Employees) is valid because the subquery computes first.
Q3What is the GROUP BY rule for SELECT columns?
Every column that appears in SELECT but is not inside an aggregate function must be listed in GROUP BY. For example: SELECT dept, city, COUNT(*) requires GROUP BY dept, city. Forgetting this causes an error in strict SQL modes. The reason: SQL doesn't know which dept or city value to show for a group if it varies within the group.
Avoid These

Common Mistakes

These errors trip up even experienced developers. Know them before they cost you in an interview.

❌
WHERE with Aggregates
Writing WHERE COUNT(*) > 5 will throw an error. WHERE cannot filter on aggregate values.
❌ WHERE COUNT(*) > 5
Use HAVING COUNT(*) > 5 after GROUP BY
❌
Missing GROUP BY Column
If you SELECT dept and city but only GROUP BY dept, the database doesn't know which city to return for each dept group.
SELECT dept, city, COUNT(*)
GROUP BY dept ← missing city!
GROUP BY dept, city — include all non-aggregate cols
❌
Confusing HAVING and WHERE
Using HAVING to filter raw rows (instead of WHERE) is technically possible but inefficient — HAVING runs after grouping, so unnecessary rows waste computation.
❌ HAVING salary > 50000
(should be WHERE)
Use WHERE for row-level filters, HAVING for group filters

The 3 Pillars

Save this mental model. These three concepts form the foundation of data analysis in SQL.

= Summary Tools
COUNT, SUM, AVG, MIN, MAX — compress many rows into one value per group
AVG(salary) → single value

= Grouping Rows
Splits table into groups by column value; aggregate runs once per group
GROUP BY dept → one row/dept

Full Query Template
SELECT column, AGG_FUNC(column) AS alias
FROM table_name
WHERE row_condition ← filters rows
GROUP BY column ← creates groups
HAVING group_condition ← filters groups
ORDER BY alias DESC ← sorts output
LIMIT 10; ← trims results
