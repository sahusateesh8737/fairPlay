01 / FOUNDATION

What Are Interview Patterns — and Why Do They Repeat?

Every SQL interview at Google, Amazon, Flipkart, or a startup is testing the same fundamental thing: can you think through a data problem and translate that thinking into a query?

Companies don't care if you memorize HAVING — they care if you know why you'd use it instead of WHERE. They repeat the same patterns because these patterns map to real business data problems every engineer encounters.

❌ What beginners think

"I need to memorize all SQL commands and practice 200 LeetCode problems."

✅ What actually works

Internalize 8 thinking patterns. Apply them to any problem. You'll recognize the pattern before reading line 2 of a question.

💡

Interview Mindset

1
Read the question and identify the pattern. Is it "find the highest X"? "Find missing Y"? "Count duplicates"? You should name the pattern in your head within 10 seconds.
2
Think in data, not in syntax. Ask: "What table gives me what? What rows do I need? What do I aggregate?"
3
Write the query skeleton first. SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT. Fill in details after.
4
Always validate with edge cases. What if no data exists? What if salary is NULL? Mentioning this earns bonus points.
02 / CLASSIC PATTERN

Second Highest Salary — The Most Common Interview Question

This is asked in 80% of SQL interviews. But most beginners fail it — not because they don't know the syntax, but because they don't think through it logically.

❌ Beginner thinking

"I'll just ORDER BY salary DESC and grab row 2." — This fails when duplicates exist or the table is empty.

✅ Interviewer thinking

"Remove the maximum salary, then find the maximum of what remains. Handle NULLs and duplicates."

The Mental Flow

All Salaries
→
Remove MAX
→
Find MAX again
→
= 2nd Highest

Method 1 — Classic Subquery (Must Know)

SELECT MAX(salary) AS SecondHighest FROM employees WHERE salary < ( -- Step 1: Find the overall maximum salary SELECT MAX(salary) FROM employees -- Step 2: Exclude it, find MAX of the rest );
1
Inner query runs first → returns 100000 (the maximum)
2
Outer query filters: WHERE salary < 100000 — all top earners removed
3
Outer MAX() on remaining rows → returns 95000 = second highest ✅

Three Alternative Methods

Method 2

LIMIT / OFFSET

SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 1;

Skip row 1 (highest), take row 2. Use DISTINCT to handle duplicates.

Method 3

Window Function

SELECT salary FROM ( SELECT salary, DENSE_RANK() OVER ( ORDER BY salary DESC ) AS rnk FROM employees ) t WHERE rnk = 2;

Most flexible. Use DENSE_RANK — it handles tied salaries correctly.

Pro Tip

N-th Highest

-- Change WHERE rnk = N -- for any N-th highest WHERE rnk = 3 -- 3rd WHERE rnk = 5 -- 5th

Window function approach generalizes to any N. Always prefer this in real interviews.

🎯

Why DENSE_RANK over RANK?

If two employees both earn ₹95,000 (rank 2 and 2), RANK() skips to 4 for next. DENSE_RANK() gives them both rank 2, next is 3. Interview-safe choice: always use DENSE_RANK().

04 / GROUP BY PATTERN

Highest Salary Per Department

This tests your understanding of GROUP BY — one of the most interview-critical concepts. The idea: instead of one answer for the whole table, give one answer per group.

The Mental Model

All Employees
→
Group by Dept
→
MAX() per group
→
One row per dept
SELECT department, MAX(salary) AS highest_salary FROM employees GROUP BY department; -- ← The magic line
1
SQL mentally splits the table into groups: one "bucket" per department.
2
MAX(salary) runs independently inside each bucket.
3
Results: one row per department, showing its maximum salary.

Advanced: Get the Employee's Name Too

SELECT e.name, e.department, e.salary FROM employees e INNER JOIN ( SELECT department, MAX(salary) AS max_sal FROM employees GROUP BY department ) d ON e.department = d.department AND e.salary = d.max_sal;

The subquery finds the max per department. The JOIN matches it back to the original row to get the employee's full name.

05 / DATA QUALITY PATTERN

Finding Duplicate Records

Data quality is a real-world problem. Finding duplicates tests your knowledge of GROUP BY + HAVING — a powerful combination that filters groups, not individual rows.

🧠

The Thinking

"If a value appears more than once, it's a duplicate. So: group by that column, count occurrences, keep only those with count > 1."

Duplicate Emails

SELECT email, COUNT(*) AS count FROM users GROUP BY email -- Group identical emails together HAVING COUNT(*) > 1; -- Only keep groups with 2+ rows

Duplicate Names + Delete

-- Find duplicates SELECT name, COUNT(*) FROM customers GROUP BY name HAVING COUNT(*) > 1; -- Delete keeping one DELETE FROM customers WHERE id NOT IN ( SELECT MIN(id) FROM customers GROUP BY name );

WHERE vs HAVING — Key Rule

⚠️ Critical Distinction

WHERE filters individual rows before grouping.
HAVING filters groups after grouping.

You CANNOT use WHERE COUNT(*) > 1 — aggregates go in HAVING.

06 / LEFT JOIN PATTERN

Missing Data — Customers Without Orders

This is a deceptively simple-sounding question that most beginners get wrong. The pattern: LEFT JOIN + WHERE IS NULL.

🧩

Why INNER JOIN fails

INNER JOIN only returns rows that match in both tables. Customers with no orders have NO match in the orders table — so they disappear. You need LEFT JOIN to keep them, then filter for those with NULL on the orders side.

Visual — The Join

customers
id:1 · Priya
id:2 · Aryan
id:3 · Sneha ← no orders
id:4 · Rajan
LEFT
JOIN
orders
cust:1 · ₹500
cust:2 · ₹800
NULL ← NULL here
cust:4 · ₹300
→
Filter
NULL
Result
id:3 · Sneha ✅
SELECT c.customer_id, c.name FROM customers c LEFT JOIN orders o ON c.customer_id = o.customer_id WHERE o.customer_id IS NULL; -- ← The magic filter
1
LEFT JOIN keeps ALL customers, even those with no matching order.
2
When no order exists, the orders columns become NULL in the result.
3
WHERE o.customer_id IS NULL filters to ONLY the customers with no orders. ✅
07 / WINDOW FUNCTION PATTERN

Running Total — Cumulative Analytics

Running totals (cumulative sums) appear in dashboards, finance reports, and growth charts. This is where Window Functions shine — you compute across rows without collapsing them.

SELECT sale_date, daily_sales, SUM(daily_sales) OVER ( ORDER BY sale_date -- ROWS UNBOUNDED PRECEDING = from start to current row ) AS running_total FROM sales;

Window Function Key Insight

Unlike GROUP BY, window functions don't collapse rows. Every original row stays in the output — it just gets an extra computed column. Think of it as "compute across a sliding window of rows."

08 / PATTERN RECOGNITION

The Master Pattern Table

Internalize this table. When you see a question, match it to a pattern in under 10 seconds.

🏆

Second / N-th Highest

Use subquery with MAX + WHERE, or DENSE_RANK() window function. Always handle NULLs.

📋

Duplicate Records

GROUP BY the column, COUNT(*), filter with HAVING COUNT(*) > 1. Never use WHERE for aggregates.

🔍

Missing / No-match Data

LEFT JOIN + WHERE right_table.id IS NULL. INNER JOIN will NOT work — it hides the missing rows.

📊

Top N Records

ORDER BY column DESC, then LIMIT N. Use DISTINCT to avoid ties. Use DENSE_RANK for ties.

🏢

Max / Avg Per Group

GROUP BY the grouping column, apply aggregate (MAX, AVG, SUM, COUNT) per group.

📈

Running Total / Rank

Use Window Functions: SUM() OVER(), ROW_NUMBER() OVER(), DENSE_RANK() OVER()

🔗

Filter After Aggregate

When you need WHERE on an aggregated value (like COUNT > 5), use HAVING, not WHERE.

⚡

Employees Above Avg

Subquery computes AVG first, outer query compares each row. Classic correlated pattern.

09 / REAL PROBLEMS

8 Scenario-Based Interview Questions

1
🏢 HR / Compensation

Find the Third Highest Salary

employees idINT PK nameVARCHAR salaryDECIMAL departmentVARCHAR
🧠 Thinking Process

Pattern: N-th highest → Use DENSE_RANK. Remove top 2 ranks, find what's left at rank 3. Handle case where 3rd highest doesn't exist (return NULL, not error).

SELECT IFNULL( (SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 2), NULL ) AS ThirdHighest;

OFFSET 2 skips first 2 rows (rank 1 and 2), LIMIT 1 takes exactly rank 3. DISTINCT handles salary ties. IFNULL handles cases with fewer than 3 distinct salaries.

2
🛒 E-commerce

Find Products That Were Never Sold

products product_idINT PK nameVARCHAR priceDECIMAL
order_items order_idINT product_idINT FK qtyINT
🧠 Thinking Process

Pattern: Missing data → LEFT JOIN + IS NULL. Keep all products (LEFT), join orders. Products with no orders will have NULL in the orders side. Filter those.

SELECT p.product_id, p.name FROM products p LEFT JOIN order_items oi ON p.product_id = oi.product_id WHERE oi.product_id IS NULL;
3
👥 CRM / Data Quality

Find Customers With Duplicate Phone Numbers

customers idINT PK nameVARCHAR phoneVARCHAR emailVARCHAR
🧠 Thinking Process

Pattern: Duplicates → GROUP BY the duplicate field (phone), COUNT, HAVING > 1. To also see which customers share the phone, self-join or use IN with subquery.

SELECT name, phone FROM customers WHERE phone IN ( SELECT phone FROM customers GROUP BY phone HAVING COUNT(*) > 1 ) ORDER BY phone;
4
🎓 Education

Top 5 Students by Score in Each Subject

scores student_idINT student_nameVARCHAR subjectVARCHAR scoreINT
🧠 Thinking Process

Pattern: Top N per group → DENSE_RANK() with PARTITION BY. LIMIT won't work here (it's global, not per-subject). Partition by subject, rank by score, filter rank ≤ 5.

SELECT student_name, subject, score, rnk FROM ( SELECT student_name, subject, score, DENSE_RANK() OVER ( PARTITION BY subject ORDER BY score DESC ) AS rnk FROM scores ) ranked WHERE rnk <= 5;
5
🏢 HR Analytics

Department With the Highest Average Salary

🧠 Thinking Process

Pattern: Max per group → GROUP BY department, compute AVG per group, ORDER BY that AVG desc, LIMIT 1 for the single top department.

SELECT department, AVG(salary) AS avg_salary FROM employees GROUP BY department ORDER BY avg_salary DESC LIMIT 1;
6
💰 Compensation

Employees Earning Above Company Average

🧠 Thinking Process

Pattern: Filter vs aggregate → Subquery computes one number (avg), outer query filters each row against it. The subquery runs once, returns a scalar value.

SELECT name, salary FROM employees WHERE salary > ( SELECT AVG(salary) FROM employees ) ORDER BY salary DESC;

The subquery (SELECT AVG(salary)...) evaluates to a single number. SQL compares each employee's salary to that one number. Elegant and fast.

7
📦 E-commerce

Customers With More Than 5 Orders

🧠 Thinking Process

Pattern: Filter on aggregate → Group by customer, count their orders, HAVING COUNT > 5. Classic HAVING use case. You could also JOIN to customers table for names.

SELECT customer_id, COUNT(*) AS order_count FROM orders GROUP BY customer_id HAVING COUNT(*) > 5 ORDER BY order_count DESC;
8
📊 Analytics

Running Sales Total by Date

🧠 Thinking Process

Pattern: Running total → Window function. SUM() OVER(ORDER BY date) accumulates from the first row to each current row. No GROUP BY needed — every row stays.

SELECT sale_date, amount, SUM(amount) OVER ( ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS running_total FROM sales ORDER BY sale_date;

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is the default but writing it explicitly shows the interviewer you understand the frame.

10 / SELF TEST

5 Tricky Interview MCQs

Q1. Which clause filters groups after GROUP BY?
A. WHERE
B. ORDER BY
C. HAVING ✓
D. FILTER
✅ HAVING filters grouped results. WHERE filters individual rows before grouping. You cannot use aggregate functions in WHERE — they go in HAVING.
Q2. You want all customers including those with NO orders. Which JOIN do you use?
A. INNER JOIN
B. LEFT JOIN ✓
C. RIGHT JOIN on orders
D. CROSS JOIN
✅ LEFT JOIN keeps all rows from the left table (customers), even when there's no match in orders. INNER JOIN would exclude customers with no orders.
Q3. Which ranking function handles ties correctly for "2nd highest" problems?
A. ROW_NUMBER()
B. RANK()
C. DENSE_RANK() ✓
D. NTILE()
✅ DENSE_RANK() assigns consecutive ranks even when there are ties (no gaps). RANK() skips numbers after ties (1,1,3). ROW_NUMBER() assigns unique numbers ignoring ties.
Q4. What does SELECT MAX(salary) FROM employees WHERE salary < (SELECT MAX(salary) FROM employees) return?
A. The highest salary
B. The second highest salary ✓
C. NULL always
D. An error
✅ The subquery returns the overall max. The outer query finds MAX from all rows below that value — which is the second highest. Classic second-highest pattern.
Q5. Which window function would you use to give each sale a rank based on amount, resetting for each region?
A. ROW_NUMBER() OVER(ORDER BY amount DESC)
B. DENSE_RANK() OVER(PARTITION BY region ORDER BY amount DESC) ✓
C. GROUP BY region, ORDER BY amount
D. MAX(amount) OVER(ORDER BY region)
✅ PARTITION BY region resets the ranking for each region. DENSE_RANK handles ties correctly. Without PARTITION BY, ranking would be global across all regions.
11 / DANGER ZONES

Common Mistakes That Cost You the Job

🚫

WHERE with Aggregate Functions

Writing WHERE COUNT(*) > 1 throws an error. Aggregates cannot go in WHERE. Always use HAVING for aggregate conditions. WHY: WHERE runs before grouping; aggregates run during/after.

🚫

INNER JOIN for Missing Data

Using INNER JOIN to find "customers without orders" returns ZERO rows — INNER JOIN only keeps matches. The correct tool is LEFT JOIN + IS NULL. This mistake is very common and very costly.

🚫

Forgetting DISTINCT in LIMIT queries

Without DISTINCT, if three employees all earn ₹95,000, you get 3 identical rows instead of one unique salary. Always use DISTINCT salary when fetching ranked salaries.

🚫

Using RANK instead of DENSE_RANK

RANK() skips numbers after ties: 1, 1, 3, 4. If two people are rank 1, there's no rank 2 — DENSE_RANK gives 1, 1, 2, 3. For N-th highest problems, DENSE_RANK is almost always correct.

🚫

Forgetting ORDER BY DESC

Writing ORDER BY salary without DESC returns the LOWEST first. For Top N / highest salary problems, you almost always want DESC (descending). Forgetting this gives completely wrong results.

🚫

NULL handling mistakes

Comparing to NULL with = NULL always fails silently. Use IS NULL or IS NOT NULL. Also, NULL < 5 is neither true nor false — it's UNKNOWN, and UNKNOWN rows are excluded.

12 / BEFORE THE INTERVIEW

Quick Revision Cards

2nd Highest Salary

  • Subquery: MAX WHERE salary < MAX
  • LIMIT/OFFSET: ORDER BY DESC LIMIT 1 OFFSET 1
  • Window: DENSE_RANK() WHERE rnk = 2
  • Always add DISTINCT for safety

Duplicates

  • GROUP BY the column
  • COUNT(*) per group
  • HAVING COUNT(*) > 1
  • NEVER use WHERE COUNT(*)

Missing Data

  • LEFT JOIN (not INNER JOIN)
  • WHERE right_side.id IS NULL
  • Customers ← JOIN → Orders
  • Keep ALL from left table

Top N Records

  • ORDER BY column DESC
  • LIMIT N
  • Per-group: PARTITION BY + DENSE_RANK
  • DISTINCT for salary-based Top N

Group Aggregates

  • GROUP BY the category column
  • MAX/AVG/SUM/COUNT per group
  • SELECT only grouped or aggregated cols
  • HAVING for post-group filters

Running Total

  • SUM() OVER(ORDER BY date)
  • No GROUP BY — rows stay intact
  • PARTITION BY for per-group running
  • Always ORDER BY in OVER()

WHERE vs HAVING

  • WHERE: before GROUP BY, row-level
  • HAVING: after GROUP BY, group-level
  • Can use both in same query
  • Aggregate in HAVING, not WHERE

JOIN Types

  • INNER: only matching rows
  • LEFT: all left + matching right
  • RIGHT: all right + matching left
  • NULL = no match on joined side
