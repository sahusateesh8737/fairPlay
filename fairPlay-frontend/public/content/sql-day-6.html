Section 06 — Very Important

ROW_NUMBER vs RANK vs DENSE_RANK

💡 Key Question

Two students tie with same marks. What happens to their rank? The answer is different for each function — and THIS is what interviewers test.

All Three — Side by Side Live Comparison

Data: 4 employees. Notice Anil & Bob both earn ₹90,000 — a tie at position 1.

NameSalaryROW_NUMBER()RANK()DENSE_RANK()
ROW_NUMBER()

Always unique. Ties get different numbers arbitrarily. Best for deduplication.

RANK()

Ties share rank. Next rank skips. Two 1st → no 2nd. Like Olympic medals.

DENSE_RANK()

Ties share rank. NO gaps. Best for "Nth highest value" queries.

Section 07 — Interview Problems

6 Real Interview Scenarios

Study the THINKING, not just the query. This is how you ace interviews.

1

Top 3 Students in Each City

Most common window function interview question

ROW_NUMBER
  • Need a rank within each city → use PARTITION BY city
  • Higher marks = lower rank number → ORDER BY marks DESC
  • Want exactly top 3 per city → use ROW_NUMBER (unique per row)
  • Window functions can't be in WHERE → wrap in CTE, then filter
SQL — Solution
WITH ranked_students AS ( SELECT name, city, marks, ROW_NUMBER() OVER( PARTITION BY city ORDER BY marks DESC ) AS rn FROM students ) SELECT name, city, marks, rn FROM ranked_students WHERE rn <= 3 ORDER BY city, rn;

⚡ Interviewer Follow-up:

If two students tie for 3rd, RANK() would give both rank 3 (returning 4 students per city). ROW_NUMBER strictly returns exactly 3. Choose based on business requirement!

2

Rank Employees by Salary in Department

HR analytics — compensation benchmarking

DENSE_RANK
  • Each department = separate window → PARTITION BY dept
  • Rank highest salary first → ORDER BY salary DESC
  • Use DENSE_RANK to handle equal salaries without gaps
SQL
SELECT name, dept, salary, DENSE_RANK() OVER( PARTITION BY dept ORDER BY salary DESC ) AS salary_rank FROM employees ORDER BY dept, salary_rank;
3

Second Highest Salary (Clean Method)

Classic trick question — DENSE_RANK is the right tool

DENSE_RANK

Why DENSE_RANK and not ROW_NUMBER? If 3 people all earn ₹90,000 (max), ROW_NUMBER would give one of them rank 2 — which is wrong. DENSE_RANK gives all of them rank 1, then the next distinct salary gets rank 2. That's the actual 2nd highest.

SQL — Works for ANY Nth highest
WITH salary_ranked AS ( SELECT name, salary, DENSE_RANK() OVER(ORDER BY salary DESC) AS dr FROM employees ) SELECT name, salary FROM salary_ranked WHERE dr = 2; -- Change to N for Nth highest
4

Running Total of Sales

Finance/analytics — cumulative revenue per region

SUM OVER
  • Separate calculation per region → PARTITION BY region
  • Accumulate in date order → ORDER BY sale_date
  • SUM with ORDER BY = running total (not partition total)
SQL
SELECT sale_date, region, amount, SUM(amount) OVER( PARTITION BY region ORDER BY sale_date ) AS running_total, AVG(amount) OVER( PARTITION BY region ) AS region_avg FROM sales ORDER BY region, sale_date;
5

Find Duplicate Records

Data cleaning — the elegant deduplication pattern

ROW_NUMBER

The Trick: Assign ROW_NUMBER partitioned by columns that define "uniqueness". Any row with rn > 1 is a duplicate. Keep rn = 1.

SQL — Duplicate Detection
WITH dupes AS ( SELECT *, ROW_NUMBER() OVER( PARTITION BY email, name ORDER BY created_at -- keep oldest record ) AS rn FROM users ) -- View duplicates SELECT * FROM dupes WHERE rn > 1; -- Delete duplicates DELETE FROM dupes WHERE rn > 1;
6

Highest Salary Per Department (With All Details)

The GROUP BY trap — window function to the rescue

RANK

❌ Wrong — loses names

SELECT dept, MAX(salary) FROM employees GROUP BY dept; -- No names!

✅ Correct — keeps all details

WITH r AS ( SELECT *, RANK() OVER( PARTITION BY dept ORDER BY salary DESC ) AS rnk FROM employees ) SELECT name, dept, salary FROM r WHERE rnk = 1;

Section 09 — Conceptual + Tricky

Interview Questions

Question 01
What is the difference between window functions and GROUP BY?
GROUP BY aggregates rows into ONE row per group — you lose individual row details. Window functions calculate across a group but keep all original rows intact. You can have an employee-level row that also shows the department aggregate. GROUP BY only shows the group aggregate row.
Question 02 — Tricky
Can you use a window function in a WHERE clause? Why or why not?
No — and this causes a very common error. SQL's order of execution is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. Window functions are evaluated in the SELECT phase, after WHERE. So you cannot reference a window function alias in WHERE. Solution: wrap in a CTE: WITH cte AS (SELECT *, ROW_NUMBER() OVER(...) AS rn FROM t) SELECT * FROM cte WHERE rn = 1
Question 03
When would you use RANK() vs DENSE_RANK()?
Use RANK() when gaps matter — like sports competitions (two 1st place means no 2nd place medal is awarded). The rank number reflects "how many people scored better than me." Use DENSE_RANK() when you want "what is this person's unique performance tier" — like finding the 2nd highest salary. DENSE_RANK is usually better for analytics queries.
Question 04 — Tricky
What does SUM() OVER() return differently WITH vs WITHOUT ORDER BY?
Without ORDER BY: Returns the total of the entire partition as a constant on every row (same number for everyone in that group).

With ORDER BY: Returns a running cumulative sum — each row shows the sum from partition start up to and including the current row. This is because ORDER BY inside OVER() implicitly adds ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.
Question 05 — Performance
Do window functions have performance implications?
Yes. Each different OVER() clause requires a separate sort pass over the data. Optimizations: (1) Combine window functions with the same PARTITION BY + ORDER BY into one pass. (2) Filter data in a CTE before applying window functions to reduce rows. (3) Ensure PARTITION BY columns are indexed on large tables. (4) In distributed systems (Spark, BigQuery), window functions trigger expensive shuffle operations — minimize partition cardinality.

Section 10

Common Mistakes

❌ Mistake 1

Confusing PARTITION BY with GROUP BY

GROUP BY physically merges rows. PARTITION BY divides logically but keeps all rows. Never use both simultaneously expecting PARTITION to override GROUP BY behavior.

-- ❌ WRONG — GROUP BY collapses rows SELECT dept, ROW_NUMBER() OVER(ORDER BY salary) FROM employees GROUP BY dept; -- ✅ RIGHT — Use PARTITION BY inside OVER() SELECT dept, ROW_NUMBER() OVER(PARTITION BY dept ORDER BY salary) FROM employees;
❌ Mistake 2

Missing ORDER BY for Ranking Functions

ROW_NUMBER(), RANK(), DENSE_RANK() without ORDER BY in OVER() produce undefined/random results. The database has no idea what order to assign numbers in.

-- ❌ WRONG — undefined result ROW_NUMBER() OVER(PARTITION BY dept) -- ✅ RIGHT ROW_NUMBER() OVER(PARTITION BY dept ORDER BY salary DESC)
❌ Mistake 3

Using Window Function in WHERE Clause

SQL processes WHERE before evaluating window functions. You'll get an error. Always wrap in CTE or subquery first.

-- ❌ WRONG — Error! SELECT *, ROW_NUMBER() OVER(...) AS rn FROM students WHERE rn <= 3; -- ✅ RIGHT — CTE pattern WITH cte AS ( SELECT *, ROW_NUMBER() OVER(...) AS rn FROM students ) SELECT * FROM cte WHERE rn <= 3;
❌ Mistake 4

Using ROW_NUMBER for "Nth Highest" Problems

If multiple rows tie for 1st place, ROW_NUMBER arbitrarily assigns one of them rank 2 — which is NOT the 2nd highest value. DENSE_RANK is the correct choice for "find the Nth highest salary" type problems.
